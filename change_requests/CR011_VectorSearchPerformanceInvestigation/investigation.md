# CR011: Vector Search Performance Investigation

## Summary
Investigation into the current implementation of vector search and memory persistence in the `SillyTavern-SceneSummariser` extension, with a focus on performance implications for long chats and backend load.

## Current Behavior

### 1. Vector Search (Retrieval)
- **Trigger**: Semantic retrieval is initiated by `filterContextInterceptor` in `src/core/injector.js`. This interceptor runs whenever SillyTavern builds a prompt for the LLM.
- **Cooldown**: A 2000ms cooldown (`RETRIEVAL_COOLDOWN`) is applied to prevent rapid-fire queries, but it still typically triggers once per user/AI message exchange.
- **Query Construction**:
    - The search text is composed of the last `semanticSearchDepth` (default 5) messages from the chat.
    - System messages are filtered out.
- **Execution**:
    - It calls `queryVectorCollection` in `src/storage/vectorHandler.js`.
    - The call is `await`ed, which blocks the prompt construction process until the vector backend (e.g., ChromaDB via Extras) returns results.
- **Injection**: Results are stored in `chatState.currentSemanticResults` and used by `buildSummaryText` to inject relevant facts into the prompt.

### 2. Vector Indexing (Persistence)
- **Trigger**: The function `persistMemoriesForChat` in `src/storage/memoryFileHandler.js` is called during:
    - Manual summary generation (`onSummariseClick` in `buttons.js`).
    - Batch summarization (`onBatchSummariseClick` in `buttons.js`).
    - Snapshot deletion/regeneration (`handleSnapshotAction`/`regenerateSnapshot` in `snapshotUI.js`).
- **Mechanism**:
    - **Full Purge**: The entire vector collection for the current chat is purged using `/api/vector/purge`.
    - **Re-indexing**: Every memory fact from every snapshot in the `chatState` is re-inserted as a new vector item.
    - **Format**: Each item's text is prefixed with the snapshot title (e.g., `Scene #1: - Fact text`).

## Performance Implications

### 1. Latency during Prompt Generation
Since the vector search is a blocking `await` within a context interceptor, any delay in the vector backend directly increases the "Time to First Token" (TTFT) for every message generated by the AI. As the chat grows and the vector database becomes larger, this latency may increase.

### 2. Inefficient Indexing (O(N) Complexity)
The "purge-and-reindex-all" strategy is extremely inefficient for larger chats:
- If a chat has 100 snapshots with 5 memories each (500 total), creating one new snapshot triggers 501 embedding generations (500 old + 1 new) and a full collection wipe.
- Most of these embeddings are redundant as the memories haven't changed.
- This puts unnecessary load on the embedding model (especially if using a remote provider like OpenAI or a local model that requires GPU resources).

### 3. Query Text Size
Using the last 5 messages as a single query string can result in a large amount of text being sent for embedding on every search. If these messages are long, the embedding quality for specific facts might be diluted, and the encoding time increases.

### 4. Redundant Searches
Even if the chat hasn't moved forward significantly, a new search is performed every 2 seconds if a prompt is being built. If the last few messages are the same, the search results will likely be identical.

## Potential Areas for Improvement (Future CRs)
- **Incremental Indexing**: Only insert new memories or update modified ones instead of purging the entire collection.
- **Asynchronous Retrieval**: Perform the vector search in parallel with other prompt construction tasks if possible, or use cached results.
- **Result Caching**: Cache search results and only re-query if the last messages in the chat have changed significantly.
- **Query Refinement**: Instead of sending all 5 messages, use a summary or extract keywords for the vector search query.
- **Collection Management**: Better handling of collection lifecycle to avoid "orphaned" collections in the vector backend.

## Conclusion
The current implementation is functional but lacks scalability for long-term usage. The O(N) re-indexing strategy and blocking retrieval are the primary bottlenecks that should be addressed in future development cycles.
